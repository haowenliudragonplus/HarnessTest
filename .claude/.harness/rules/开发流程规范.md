# 开发流程规范

> 一次完整需求的 10 阶段流水线 + 三 Agent 编排规则。所有 Agent 必读。

---

## 10 阶段流水线总览

| # | 阶段 | 负责 Agent | MVP 状态 | 产出物 |
|---|---|---|---|---|
| 1 | 需求分析 | Planner | ✅ | `request_analysis/spec.md` + `tasks.md` |
| 2 | 需求评审 | Evaluator (Plan) | ✅ | `request_analysis/review/spec_review_v{N}.md` |
| 3 | 编码实现 | Generator | ✅ | 代码改动 + `coding/coding_report_v{N}.md` |
| 4 | 编码评审 | Evaluator (Code) | ✅ | `coding/review/code_review_v{N}.md` |
| 5 | 单元测试编写 | —（Phase 2） | ⏭️ | — |
| 6 | 单元测试评审 | —（Phase 2） | ⏭️ | — |
| 7 | 代码提交 | **用户手动** | ✅ | — |
| 8 | 本地/CI 验证 | **用户手动** | ✅ | — |
| 9 | 部署验证 | **用户手动** | ✅ | — |
| 10 | 用户最终 ACK | **用户** | ✅ | `summary.md` 标 ACK |

---

## 自动 vs 人工边界

### ✅ 主会话自动接力（无需用户指令）
- Stage 1 → 2：Planner 完成后自动调 Evaluator(Plan)
- Stage 2 → 2'（返工）：Evaluator REVISION_REQUIRED 且未超限 → 自动调 Planner-revise
- Stage 2' → 2：Planner 返工完成 → 自动再评审
- Stage 3 → 4：Generator 完成后自动调 Evaluator(Code)
- Stage 4 → 4'（返工）：同上
- Stage 4' → 4：同上

### ⏸ HITL 暂停等用户（仅 2 个点）
- **HITL #1**：Stage 2 APPROVED → 暂停展示 spec + review 摘要，等用户回复"确认"才进入 Stage 3
- **HITL #2**：Stage 4 APPROVED → 暂停展示 diff + review 摘要，等用户回复"确认"才进入 Stage 7

### 🚨 升级人工（异常路径）
- Stage 2 第 3 轮仍 MUST FIX → 主会话停止自动，向用户汇报
- Stage 4 第 2 轮仍 MUST FIX → 同上
- 任何 Agent 输出 `🚨 BLOCKED` → 同上

### 👤 完全人工
- Stage 7（代码提交，由用户自行完成）
- Stage 8（本地/CI 实际运行验证）
- Stage 9（部署 / 灰度发布）

## 循环上限

| 评审类型 | 上限 | 超限处理 |
|---|---|---|
| Plan Review | 3 轮（v1, v2, v3） | ESCALATE_TO_HUMAN |
| Code Review | 2 轮（v1, v2） | ESCALATE_TO_HUMAN |

判定 N：变更目录下已存在的 review 文件数。新一轮 review 命名为 `*_review_v{N+1}.md`。

## 状态机的 11 条规则（与 CLAUDE.md 一致）

| # | 当前状态 | 下一步 |
|---|---|---|
| S1 | 新需求，无变更目录 | 创建变更目录，调 Planner |
| S2 | spec.md + tasks.md 存在，无 spec_review_v*.md | 调 Evaluator(Plan) |
| S3 | spec_review_v{N}.md REVISION_REQUIRED，N<3 | 调 Planner-revise |
| S4 | spec_review_v{N}.md REVISION_REQUIRED，N≥3 | 🚨 ESCALATE |
| S5 | spec_review_v{N}.md APPROVED，无 coding_report | ⏸ HITL #1 |
| S6 | 用户确认 HITL #1 | 调 Generator |
| S7 | coding_report_v{N}.md 存在，无 code_review_v{N}.md | 调 Evaluator(Code) |
| S8 | code_review_v{N}.md REVISION_REQUIRED，N<2 | 调 Generator-revise |
| S9 | code_review_v{N}.md REVISION_REQUIRED，N≥2 | 🚨 ESCALATE |
| S10 | code_review_v{N}.md APPROVED | ⏸ HITL #2 |
| S11 | 用户确认 HITL #2 | 输出人工 checklist，更新 summary.md，流程结束 |

## 5 个文章原 HITL 在本 MVP 的处理

| 文章原 HITL | 本 MVP 处理 |
|---|---|
| 1. 需求待决议确认 | 合并到 HITL #1（用户在确认 spec 时一并解决待决议） |
| 2. 计划评审后确认 | = HITL #1 |
| 3. 编码评审后确认 | = HITL #2 |
| 4. 部署环境参数确认 | 在 Stage 11 输出的人工 checklist 中提示 |
| 5. 最终交付确认 | = Stage 10（用户主动 ACK 即可，主会话不强制等待） |

## summary.md 维护规范

每次变更目录创建时，主会话写入空模板。每阶段完成后，**主会话**（而非 Agent）追加一行：

```markdown
# 变更摘要：{type}-{name}-{YYYYMMDD}

## 基本信息
- 创建时间：YYYY-MM-DD HH:MM
- 原始需求：<用户原话>
- 状态：[进行中 | 已交付 | 已升级人工]

## 阶段执行记录

| 时间 | 阶段 | 执行者 | 结果 | 产出物 |
|---|---|---|---|---|
| HH:MM | 1.需求分析 | Planner | ✅ | spec.md / tasks.md |
| HH:MM | 2.需求评审 v1 | Evaluator | APPROVED | spec_review_v1.md |
| HH:MM | HITL #1 | 用户 | ✅ 确认 | — |
| HH:MM | 3.编码 | Generator | ✅ | coding_report_v1.md |
| ... | ... | ... | ... | ... |

## 最终 ACK
- 用户 ACK 时间：YYYY-MM-DD HH:MM
- 用户 ACK 备注：—
```

**禁止追加重复行**（防文章 5.1 节"摘要因追加倾向出现重复"问题）。如发现已有相同阶段记录，覆盖而非追加。

---

## 变更目录命名

格式：`{type}-{name}-{YYYYMMDD}`

| 字段 | 取值 |
|---|---|
| type | `feat` / `fix` / `chore` / `refactor` |
| name | 短横线分隔小写 kebab-case，5 个词以内 |
| date | 创建当日 YYYYMMDD |

举例（命名形式示意）：
- `feat-<short-name>-YYYYMMDD`
- `fix-<short-name>-YYYYMMDD`
- `refactor-<short-name>-YYYYMMDD`
