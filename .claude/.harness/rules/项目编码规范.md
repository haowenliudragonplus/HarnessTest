# 项目编码规范（不变约束）

> Generator 编码时必读、Evaluator 评审时逐条检查。违反任一条 → MUST FIX。
>
> 这些规则是**机器可验证**的硬约束——文章 5.2 节："If it can't be mechanically enforced, the agent will drift."

---

## 1. 命名前缀（强约束）

| 元素类型 | 前缀 | 范例 |
|---|---|---|
| 业务模块（继承 ModuleBase） | `Mod*` | `ModAccount`, `ModBag`, `ModTip` |
| FSM 状态（继承 FsmStateBase） | `FsmState_*` | `FsmState_Login`, `FsmState_Home` |
| UI 视图 | `UIView_*` | `UIView_Boot`, `UIView_Notice` |
| 启动流水线步骤 | `Procedure*` | `ProcedureInitialize`, `ProcedureLoadDll` |
| 配表类 | `Table_*` | `Table_Common_GlobalReward` |
| 常量集合 | `Const_*` | `Const_Boot` |
| 事件结构 | `Evt*` | `EvtBindFacebook`, `EvtBindApple` |
| UI 名常量 | `UIViewName.*` 中 | `UIViewName.UIView_Notice` |

---

## 2. 数值类型（业务敏感）

### 必须用 `long` 的字段
- 价格、货币、钻石、金币、道具数量、经验、能量等所有"游戏内可累积的数值"
- 时间戳（毫秒级 / 秒级，看 `TimeUtils.GetServerTimeStamp()` 返回 long）

### 禁用
- ❌ `float` / `double` 表示货币或道具数量（IEEE 754 精度问题会导致丢钱/丢道具）
- ❌ `int` 表示可能溢出的数量（如经验/积分总额，玩家活跃时易超 21 亿）

### 例外
- 概率、缩放系数、UI 动画时长等"非业务关键"的小数仍可用 float
- 配表中物理坐标可以 float

---

## 3. 模块访问统一入口

### 唯一允许的访问方式
```csharp
Game.GetMod<ModAccount>().Login();
Game.GetMod<ModFsm>().ChangeState<FsmState_Home>();
Game.GetMod<ModBag>().AddItem(ids, counts, reason);
```

### 禁止
- ❌ 直接 `new ModXxx()`（除框架初始化处）
- ❌ 通过单例 `ModXxx.Instance`（除非该 Module 显式定义且有充分理由）
- ❌ 静态字段长期持有 Module 引用（Module 可能被 Dispose）

---

## 4. 资源加载

### 唯一允许
```csharp
var prefab = await Game.GetMod<ModAsset>().LoadAsync<GameObject>(path);
```

### 禁用（除 Boot 阶段外）
- ❌ `Resources.Load<T>(path)`
- ❌ `AssetBundle.LoadFromFile / LoadFromMemory`
- ❌ `YooAssets.LoadAssetSync/Async`（应通过 ModAsset 封装）

### Boot 阶段例外
[Boot.cs:39](../../../Assets/Scripts/Main/Boot.cs#L39) 处 `Resources.Load<GameObject>(Const_Boot.PrefabPath_BootView)` 是允许的——彼时 ModAsset 尚未启动。

---

## 5. 异步与并发

### 唯一允许
- `async UniTask Foo()` + `await`
- 模块间异步交互全部用 UniTask

### 禁用
- ❌ `StartCoroutine` 跨模块传递（会和 MonoBehaviour 生命周期耦合）
- ❌ `async void` 写公开方法（除 MonoBehaviour 事件回调如 `async void Start()`）
- ❌ `Thread` / `Task.Run`（除明确的纯计算密集场景）

### 例外
- `Start()`、`Update()` 等 MonoBehaviour 事件可用 `async void`
- 第三方 SDK 回调使用 `Action<bool, string>` 模式（参考 [ModAccount.OnBindFbResult](../../../Assets/Scripts/Hotfix/Common/Module/Account/ModAccount.cs)）

---

## 6. 日志

### 唯一允许
```csharp
CLog.Info("xxx");
CLog.Warn("xxx");
CLog.Error("xxx");
```

### 禁用
- ❌ `Debug.Log` / `Debug.LogError` / `Debug.LogWarning`
- ❌ `Console.WriteLine`
- ❌ `print(...)`

---

## 7. 主工程 ↔ 热更工程边界

### 主工程
- 只允许：MonoBehaviour 类、SDK 初始化、HybridCLR 启动相关代码、桥接 `Loader.OnXxx` 委托
- **绝不**：业务逻辑、`new` 热更类、引用 `Game.GetMod<>`

### 热更工程
- 只允许：业务模块、FSM、UI、配表
- 通过 `Loader.OnUpdate += ...` 接收主工程生命周期事件
- **绝不**：继承 MonoBehaviour（HybridCLR 限制）

---

## 8. 第三方 SDK 调用

### 必须
- 封装在专用 Module（Adapter 层）内，例如 IAP 在 `ModIAP`，广告在 `ModAdv`
- 调用必须设**超时**（一般 ≤30s）
- 必须设**降级方案**（SDK 不可用时给玩家明确提示，不能默默卡死）
- 编辑器中走 mock（参考 [ModAccount.BindFacebook](../../../Assets/Scripts/Hotfix/Common/Module/Account/ModAccount.cs) 的 `#if UNITY_EDITOR` 分支）

### 禁用
- ❌ 在业务模块（如 ModBag / ModFsm）内直接 `SDK<IXxx>.Instance.Foo()`——必须经过 Adapter
- ❌ 不设超时的同步等待

---

## 9. 事件系统

### 推荐
```csharp
// 派发
Game.GetMod<ModEvent>().Dispatch(new EvtBindFacebook(success));

// 监听（在 ModuleBase.RegisterEvent 中）
public override void RegisterEvent()
{
    Listen<EvtBindFacebook>(OnBindFacebook);
}
```

### 禁用
- ❌ 跨模块直接函数调用并互相 Save 引用（紧耦合）
- ❌ 用 `static event` 做跨模块通信

---

## 10. UI 打开

### 唯一允许
```csharp
var openData = new UIView_Notice.OpenData { content = "...", showMidBtn = true };
Game.GetMod<ModUI>().OpenSync(UIViewName.UIView_Notice, openData);

// 或异步
await Game.GetMod<ModUI>().OpenAsync(UIViewName.UIView_Reward, openData);
```

### 禁用
- ❌ `Instantiate(uiPrefab)` 手动创建 UI（绕过 ModUI 生命周期）
- ❌ 直接通过 `GameObject.Find` 操作 UI 树

---

## 11. 存档读写

### 唯一允许
```csharp
var storage = SDK<IStorage>.Instance.Get<StorageClientCommon>();
storage.UserData.FirstLoginTime = TimeUtils.GetServerTimeStamp();
```

### 禁用
- ❌ `PlayerPrefs.SetXxx`（绕过 Storage 模块，无法跨设备同步）
- ❌ 自定义文件读写

---

## 12. 文件级约束

### 禁止的"顺手重构"（防熵累积）
- ❌ tasks.md 范围之外的文件改动
- ❌ 删除/重命名"看起来没用"的方法（可能有热更被引用）
- ❌ 重排 using / 调整空白行（除非已经触及该文件其他改动）

### 允许的"顺手"
- ✅ 修复同一个方法内的明显笔误（typo、错误的 nameof）
- ✅ 补充缺失的 null 检查（如果不补就会 NRE）

---

## 13. 注释规范

### 必须加注释的情况
- 业务约束的隐含原因（"价格用 long 是因为......"）
- 与文档不一致的实际行为
- 短期 hack / TODO（必须标 TODO 加日期）

### 默认不加注释
- 方法名 / 变量名已能自解释的逻辑
- 显而易见的循环 / 条件
- "做了什么"的解释（用更好的命名替代）
